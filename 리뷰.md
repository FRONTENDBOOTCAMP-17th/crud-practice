## 1. 프로젝트 개요

Fetch API를 활용한 게시판 CRUD 웹 애플리케이션입니다. 회원가입, 로그인, 게시글 작성/목록/상세/수정/삭제 기능을 구현했습니다.

### 파일 구성

| 파일 | 역할 |
|------|------|
| `register.html` | 회원가입 (입력값 검증 포함) |
| `login.html` | 로그인 (JWT 저장) |
| `index.html` | 게시글 목록 |
| `write.html` | 게시글 작성 (draft → publish) |
| `detail.html` | 게시글 상세 + 수정/삭제 버튼 |
| `editPost.html` | 게시글 수정 |

## 2. 잘한 점

### 2.1 완전한 CRUD 구현
- Create(작성), Read(목록/상세), Update(수정), Delete(삭제) 모든 기능이 구현되어 있습니다.
- 다른 팀 프로젝트와 비교했을 때 수정/삭제까지 완성한 점이 돋보입니다.

### 2.2 입력값 검증 (register.html)
```javascript
const rules = {
  username: /^[a-zA-Z0-9]{4,20}$/,
  password: /^\S{4,20}$/,
  nickname: /^[a-zA-Z0-9ㄱ-ㅎㅏ-ㅣ가-힣]{2,20}$/,
};
```
- 정규식으로 아이디/비밀번호/닉네임 형식을 검증합니다.
- placeholder에 규칙을 안내하는 점도 좋습니다.

### 2.3 내 글 판별 로직 (detail.html)
```javascript
const userCheck = await fetch(".../posts/my", { ... });
// 내 게시글 목록에서 현재 글이 있는지 확인
for (let i = 0; i < data.length; i++) {
  if (data[i].id === Number.parseInt(postId)) {
    isMyPost = true;
    break;
  }
}
```
- 내가 쓴 글에만 수정/삭제 버튼을 표시하는 권한 처리가 잘 되어 있습니다.

### 2.4 일관된 UI 스타일링
- `index.html`, `write.html`, `detail.html`, `editPost.html`에 통일된 CSS 스타일이 적용되어 있습니다.
- 카드형 레이아웃, hover 효과, 반응형 max-width 등 완성도가 높습니다.

### 2.5 UX 세부 사항
- 삭제 전 `confirm()` 확인 대화상자를 사용합니다.
- `location.replace()`를 사용하여 뒤로가기 시 폼 재전송을 방지합니다.
- `white-space: pre-wrap`으로 게시글 내용의 줄바꿈을 유지합니다.

### 2.6 Git 브랜치 활용
- 4명의 팀원이 각자 브랜치(`jongho`, `seonwoo`, `youngeun`, `yuna`)를 만들어 작업했습니다.
- PR을 통해 main에 머지하는 워크플로우를 사용했습니다.

## 3. 개선이 필요한 점

### 3.1 XSS 취약점 — 심각

`index.html`과 `detail.html`에서 API 응답 데이터를 `innerHTML`로 직접 삽입하고 있습니다.

```javascript
// index.html 108행
item.innerHTML = `
  <div class="post-title">${post.title}</div>
  <div class="post-preview">${post.contentPreview}</div>
`;

// detail.html 127행
postEl.innerHTML = `
  <div class="post-content">${post.content}</div>
`;
```

누군가 게시글 제목이나 내용에 `<script>alert('해킹')</script>` 같은 코드를 넣으면 다른 사용자의 브라우저에서 실행됩니다.

**해결 방법**: `textContent`를 사용하거나, DOM 요소를 `createElement`로 생성합니다.

```javascript
// 안전한 방법
const titleEl = document.createElement("div");
titleEl.className = "post-title";
titleEl.textContent = post.title;  // HTML이 해석되지 않음
```

### 3.2 API 키 불일치 (editPost.html)

```javascript
// editPost.html 133행 — 다른 API 키 사용
"https://api.fullstackfamily.com/api/edu/ws-dbd822/posts/" + postId

// 나머지 모든 파일
"https://api.fullstackfamily.com/api/edu/ws-1c07e0/..."
```

`editPost.html`만 `ws-dbd822` 키를 사용하고 있어서 수정 기능이 동작하지 않을 수 있습니다.

### 3.3 localStorage를 통한 데이터 전달 (detail → editPost)

```javascript
// detail.html — 수정 버튼 클릭 시
localStorage.setItem("title", post.title);
localStorage.setItem("content", post.content);
window.location.replace(`/editPost.html?id=${postId}`);

// editPost.html — localStorage에서 읽기
title.value = localStorage.getItem("title");
content.value = localStorage.getItem("content");
```

문제점:
- 사용자가 수정 페이지에 직접 접근하면 이전 데이터가 남아있을 수 있습니다.
- 여러 탭에서 동시에 다른 글을 수정하면 데이터가 충돌합니다.

**해결 방법**: `editPost.html`에서 `postId`로 API를 호출하여 데이터를 직접 가져오는 것이 안전합니다.

### 3.4 절대 경로 문제 (detail.html)

```javascript
// detail.html 144행
window.location.replace(`/editPost.html?id=${postId}`);
//                        ^ 루트 절대 경로
```

`/editPost.html`은 도메인 루트 기준이므로, GitHub Pages나 하위 폴더에 배포하면 404 에러가 발생합니다. 상대 경로 `editPost.html`을 사용해야 합니다.

### 3.5 로그인 후 페이지 이동 없음 (login.html)

```javascript
localStorage.setItem("token", data.data.token);
result.textContent = "로그인 성공!\n토큰값: " + data.data.token;
// 로그인 후 페이지 이동이 없음
```

로그인 성공 후 토큰 값만 화면에 표시하고, 게시판 목록(`index.html`)으로 이동하지 않습니다. `location.href = "index.html"` 추가가 필요합니다.

### 3.6 loadMyInfo 에러 처리 미흡 (write.html, editPost.html)

```javascript
async function loadMyInfo() {
  try {
    const res = await fetch(".../auth/me", { ... });
    // res.ok 검사 없음
    const { data } = await res.json();
    document.getElementById("nickname").textContent = "작성자: " + data.nickname;
  } catch (error) {
    console.error("에러:", error.message);  // 콘솔에만 출력
  }
}
```

- `res.ok` 검사가 없어서 401(토큰 만료) 응답 시 `data`가 undefined가 될 수 있습니다.
- 에러가 콘솔에만 출력되어 사용자가 문제를 인지할 수 없습니다.

### 3.7 글쓰기 버튼 항상 노출 (index.html)

```html
<a href="write.html" class="btn-write">글쓰기</a>
```

비로그인 상태에서도 "글쓰기" 버튼이 보입니다. `write.html`에서 토큰 검사를 하지만, 버튼 자체를 숨기거나 로그인 페이지로 안내하는 것이 더 나은 UX입니다.
